# Dev Log — 2026-04-15

## What we did

### Architecture decision: Python for Windows automation
Decided to use a Python subprocess for all Windows system interaction rather than
calling Win32/UIA APIs directly from Node.js via FFI. Python's pywin32 and comtypes
libraries handle Win32 and COM (which UIA uses) far more cleanly than koffi would
from TypeScript. TypeScript remains the brain (LLM, tool dispatch, conversation loop),
Python is the hands (OS interaction).

Communication between the two is JSON over stdio — TypeScript spawns the Python
process and they exchange newline-delimited JSON on stdin/stdout.

### Folder structure created
Two new directories added alongside the existing `core/`:

**`win/`** — TypeScript bridge layer
- `index.ts` — public API re-exports for the module
- `bridge.ts` — spawns and manages the Python subprocess, owns the stdio channel
- `tools.ts` — typed wrappers: click(), typeText(), focusWindow(), findElement()
- `snapshot.ts` — requests world state from Python, returns typed 5.1 schema object
- `events.ts` — receives proactive event pushes from Python, re-emits as EventEmitter

**`python/`** — Windows automation layer
- `win_agent.py` — entry point, stdin/stdout dispatch loop, COM initialization
- `uia.py` — Windows UI Automation: enumerate windows, walk element trees to depth 5
- `win32.py` — Win32 API: window lookup, foreground, synthetic input, geometry
- `snapshot.py` — combines uia + win32 into a 5.1 schema JSON world state snapshot
- `events.py` — background thread monitoring focus changes and process start/stop

### Return type conventions established
- Internal helpers (used only within their file): raw primitives — `int` for HWNDs, `list[int]`
- Module-boundary query functions: `dict` with named fields
- Action functions (click, keystroke, focus): `{"ok": bool, "error"?: str}`
- Snapshot: full `dict` matching the 5.1 schema

### Implemented `get_window_info(hwnd)` in `win32.py`
First concrete implementation. Given an HWND returns:
```python
{
    "hwnd": int,
    "title": str,           # may be empty string for titleless windows
    "rect": {
        "x": int, "y": int,
        "width": int, "height": int
    },
    "pid": int,
    "process_name": str | None,   # None if process exited between calls
    "visible": bool
}
```
`GetWindowRect` returns `(left, top, right, bottom)` — converted to x/y/width/height.
`IsWindowVisible` result cast to bool so JSON serializes as true/false not 0/1.
`process_name` resolved via `psutil` — falls back to `None` rather than raising if the
process has already exited by the time we query it.

Also implemented `get_windows_from_pid(target_pid)` which enumerates all visible
windows belonging to a PID and returns a list of `get_window_info` dicts.

### Error handling convention for `win32.py`
All functions raise `RuntimeError` with a descriptive message on failure rather than
returning sentinel values (`False`, `None`, empty list). This is intentional for agent
use — the error string lands in the `tool_result` content block and the agent can reason
about it directly. Sentinel values give the agent nothing to act on.

Each `RuntimeError` wraps the underlying `pywintypes.error` (preserving the Win32 error
code via `from e`) but exposes a clean human-readable message including the relevant
input (e.g. `hwnd=80396`) so the agent can decide whether to retry, re-query, or abort.

Fields that are *sometimes* inaccessible (e.g. `exe`, `cmdline`, `username` on elevated
processes) fall back to `None` rather than raising — the dict is always fully populated.

### Window enumeration functions
Two discovery functions added for agent use:

**`get_all_visible_windows()`** — primary agent discovery tool. Returns every visible,
titled top-level window. Filters out untitled background windows (shell components, IME
hosts, tooltip anchors) that have no interactive surface. Call this first to find a
window before calling `set_foreground_window` or any HWND-based function.

**`get_all_windows()`** — unfiltered enumeration. Returns everything `EnumWindows` sees
including invisible and untitled windows. Used for diagnostics or when the full picture
is needed.

Both return the same `get_window_info` dict shape including `process_name`, so the agent
can identify windows without cross-referencing a separate process table.

### `get_process_info(pid)` added
Returns a rich snapshot of process runtime state:
```python
{
    "pid": int,
    "name": str,            # executable name e.g. "chrome.exe"
    "exe": str | None,      # full path — None if access denied
    "cmdline": list | None, # full argv — key for distinguishing instances
    "status": str,          # running | sleeping | stopped | zombie
    "username": str | None, # domain\user — None if access denied
    "ppid": int,
    "created": str,         # ISO 8601 timestamp
    "cpu_percent": float,   # sampled over 0.1s interval
    "memory_mb": float,     # resident set size
    "num_threads": int,
}
```
`cmdline` is the primary field for distinguishing multiple instances of the same binary
(e.g. 30+ `chrome.exe` entries in the PID table).

### `set_foreground_window(hwnd)` added
Wraps `SetForegroundWindow`. Raises `RuntimeError` on invalid/stale handle or focus
denial (UAC elevation mismatch between caller and target process).

### Titleless windows
Many windows a process owns have no title — message-only windows, drag-and-drop
helpers, IME windows, hidden framework windows. These are expected and not a bug.
Decision: keep `win32.py` as a faithful low-level wrapper, `get_all_visible_windows`
filters by title for agent use, `get_all_windows` exposes everything for diagnostics.
