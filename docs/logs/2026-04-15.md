# Dev Log — 2026-04-15

## What we did

### Architecture decision: Python for Windows automation
Decided to use a Python subprocess for all Windows system interaction rather than
calling Win32/UIA APIs directly from Node.js via FFI. Python's pywin32 and comtypes
libraries handle Win32 and COM (which UIA uses) far more cleanly than koffi would
from TypeScript. TypeScript remains the brain (LLM, tool dispatch, conversation loop),
Python is the hands (OS interaction).

Communication between the two is JSON over stdio — TypeScript spawns the Python
process and they exchange newline-delimited JSON on stdin/stdout.

### Folder structure created
Two new directories added alongside the existing `core/`:

**`win/`** — TypeScript bridge layer
- `index.ts` — public API re-exports for the module
- `bridge.ts` — spawns and manages the Python subprocess, owns the stdio channel
- `tools.ts` — typed wrappers: click(), typeText(), focusWindow(), findElement()
- `snapshot.ts` — requests world state from Python, returns typed 5.1 schema object
- `events.ts` — receives proactive event pushes from Python, re-emits as EventEmitter

**`python/`** — Windows automation layer
- `win_agent.py` — entry point, stdin/stdout dispatch loop, COM initialization
- `uia.py` — Windows UI Automation: enumerate windows, walk element trees to depth 5
- `win32.py` — Win32 API: window lookup, foreground, synthetic input, geometry
- `snapshot.py` — combines uia + win32 into a 5.1 schema JSON world state snapshot
- `events.py` — background thread monitoring focus changes and process start/stop

### Return type conventions established
- Internal helpers (used only within their file): raw primitives — `int` for HWNDs, `list[int]`
- Module-boundary query functions: `dict` with named fields
- Action functions (click, keystroke, focus): `{"ok": bool, "error"?: str}`
- Snapshot: full `dict` matching the 5.1 schema

### Implemented `get_window_info(hwnd)` in `win32.py`
First concrete implementation. Given an HWND returns:
```python
{
    "hwnd": int,
    "title": str,           # may be empty string for titleless windows
    "rect": {
        "x": int, "y": int,
        "width": int, "height": int
    },
    "pid": int,
    "visible": bool
}
```
`GetWindowRect` returns `(left, top, right, bottom)` — converted to x/y/width/height.
`IsWindowVisible` result cast to bool so JSON serializes as true/false not 0/1.

Also implemented `get_windows_from_pid(target_pid)` which enumerates all visible
windows belonging to a PID and returns a list of `get_window_info` dicts.

### Titleless windows
Many windows a process owns have no title — message-only windows, drag-and-drop
helpers, IME windows, hidden framework windows. These are expected and not a bug.
Decision: keep `win32.py` as a faithful low-level wrapper (no title filtering),
let `snapshot.py` decide what the LLM needs to see.
